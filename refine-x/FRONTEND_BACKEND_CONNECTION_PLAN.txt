PIPELINE (Celery background processing):
  Phase 1 (process_csv_file):
    Load CSV → Global Rules → Struct Rules → Column Relevance Gate (GPT-4o)
    → status = "awaiting_review" → STOP (waits for user column confirmation)

  Phase 2 (resume_pipeline_after_review):
    Replay Steps 1-4 → Apply column filter → AI Classification → HTYPE
    Detection → Tiered Merge → Sessions 4-16 cleaning → Quality Score
    → Analytical Formulas → Persist → status = "completed"

AUTH MECHANISM:
  JWT Bearer tokens. Every protected endpoint requires:
    Authorization: Bearer <access_token>
  Token created by POST /auth/login, validated by get_current_user dependency.
  Token payload: { sub: "<user_id>" }
  Expiry: 30 minutes (configurable via ACCESS_TOKEN_EXPIRE_MINUTES)

LAYER 1: API CLIENT (create first, everything depends on it)
───────────────────────────────────────────────────────────────────────────

Create: lib/api/client.ts

  Purpose: Centralized HTTP client that:
    • Points to NEXT_PUBLIC_API_URL (default: http://localhost:8000)
    • Automatically attaches Authorization: Bearer <token> header
    • Handles 401 responses by redirecting to /auth/login
    • Handles network errors gracefully
    • Provides typed response parsing

  Implementation approach:
    const API_URL = process.env.NEXT_PUBLIC_API_URL || "http://localhost:8000"

    async function apiClient<T>(
      endpoint: string,
      options?: RequestInit & { skipAuth?: boolean }
    ): Promise<T> {
      const token = typeof window !== "undefined"
        ? localStorage.getItem("access_token") : null

      const headers: Record<string, string> = {
        "Content-Type": "application/json",
        ...options?.headers as Record<string, string>,
      }

      if (token && !options?.skipAuth) {
        headers["Authorization"] = `Bearer ${token}`
      }

      const res = await fetch(`${API_URL}${endpoint}`, {
        ...options,
        headers,
      })

      if (res.status === 401) {
        localStorage.removeItem("access_token")
        window.location.href = "/auth/login"
        throw new Error("Unauthorized")
      }

      if (!res.ok) {
        const error = await res.json().catch(() => ({ detail: "Unknown error" }))
        throw new Error(error.detail || `HTTP ${res.status}`)
      }

      return res.json()
    }

  SPECIAL CASE — Login endpoint:
    The backend uses OAuth2PasswordRequestForm, which expects
    application/x-www-form-urlencoded body, NOT JSON.
    The login function must use URLSearchParams:

    async function login(email: string, password: string) {
      const body = new URLSearchParams({ username: email, password })
      const res = await fetch(`${API_URL}/auth/login`, {
        method: "POST",
        headers: { "Content-Type": "application/x-www-form-urlencoded" },
        body,
      })
      // ... parse Token response
    }

  SPECIAL CASE — File upload:
    The upload endpoint expects multipart/form-data with the file.
    Must NOT set Content-Type header (let browser set boundary):

    async function uploadFile(file: File) {
      const formData = new FormData()
      formData.append("file", file)
      const token = localStorage.getItem("access_token")
      const res = await fetch(`${API_URL}/upload`, {
        method: "POST",
        headers: { Authorization: `Bearer ${token}` },
        body: formData,
      })
      // ... parse UploadJobResponse
    }

  Purpose: React context that provides:
    • user: UserResponse | null
    • token: string | null

    if yes, call GET /auth/me to validate it →

PART 7: CRITICAL NOTES & GOTCHAS
═══════════════════════════════════════════════════════════════════════════

1. LOGIN USES FORM-ENCODED BODY, NOT JSON
   The backend's /auth/login uses OAuth2PasswordRequestForm which expects
   Content-Type: application/x-www-form-urlencoded
   Body: username=<email>&password=<password>
   NOT: { "email": "...", "password": "..." }
   The field is called "username" even though it's an email.

2. FILE UPLOAD USES MULTIPART FORM DATA
   POST /upload expects multipart/form-data with field name "file".
   Do NOT set Content-Type header manually — let the browser set it
   with the correct boundary.

3. CHART DATA IS ALREADY RECHARTS-COMPATIBLE
   The backend's Chart model stores data as JSON arrays that are
   formatted for direct use with Recharts. No transformation needed.
   Just pass chart.data directly to <LineChart data={chart.data}>.

4. THE PIPELINE HAS A HUMAN-IN-THE-LOOP STEP
   After upload, the pipeline pauses at "awaiting_review" status.
   The frontend MUST implement the column review UI:
   - Poll status until awaiting_review
   - Show column_relevance_result (GPT-4o's recommendations)
   - Let user check/uncheck columns
   - POST /upload/jobs/{id}/review with confirmed_columns
   - Continue polling until "completed"
   Without this, the pipeline never finishes.

5. CORS IS ALREADY CONFIGURED
   main.py already has:
   allow_origins=["http://localhost:3000"]
   No proxy configuration needed.

6. MANY FRONTEND PAGES ARE SERVER COMPONENTS
   Analyses, Datasets, Insights, Visualizations, History are all
   server components (no 'use client'). To add hooks (useState,
   useEffect, useAuth), they must be converted to client components
   by adding 'use client' at the top.

7. THE FRONTEND HAS NO CONCEPT OF "JOB ID"
   The current URL structure uses /dashboard/project/:id but the
   backend uses job IDs. The mapping is: project ID = upload job ID.
   The new project flow should pass the job ID through URL search
   params: /dashboard/project/new/profile?jobId=123

8. JWT EXPIRY IS 30 MINUTES
   Tokens expire in 30 minutes. The frontend should:
   - Catch 401 errors globally in the API client
   - Clear token and redirect to login
   - Consider adding a token refresh mechanism (backend doesn't
     have a refresh endpoint yet, but could be added)

9. BACKEND RUNS ON 4 PROCESSES
   You need: PostgreSQL + Redis + MinIO (via docker-compose),
   then: uvicorn (API), celery worker, and optionally celery beat.
   All must be running for the full pipeline to work.

10. THE ONBOARDING WIZARD PARTIALLY MAPS TO BACKEND
    Step 1 (org type) → No backend endpoint
    Step 2 (workspace) → No backend endpoint  
    Step 3 (file upload) → POST /upload (this works!)
    For MVP, only wire Step 3. Steps 1-2 can store in localStorage.
